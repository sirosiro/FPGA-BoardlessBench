# Verilator とは
[Verilator](https://www.veripool.org/verilator/)（ベリレータ）は一言で言えば **「世界最速級のオープンソース・オープンソース HDL（ハードウェア記述言語）シミュレータ」** です。

特に、C++やSystemCに変換してシミュレーションを行う「サイクル精度（Cycle-accurate）」のシミュレータとして、半導体設計や組込みシステム開発の現場で非常に高く評価されています。

その特徴や仕組みを解説します。

---

## 1. Verilator の主な特徴

Verilator は、Icarus Verilog などの他のオープンソースシミュレータとは一線を画す特徴を持っています。

* **圧倒的な実行速度:** Verilog コードを高度に最適化された C++ コードに変換（コンパイル）してから実行するため、非常に高速です。商用シミュレータと比較しても、多くのケースで数倍から十数倍のパフォーマンスを発揮します。
* **サイクル精度（Cycle-accurate）:**
    イベントドリブン方式（タイミング重視）ではなく、クロックエッジごとの状態変化のみを追う「サイクル精度」のシミュレータです。ゲートレベルの詳細な遅延（ピコ秒単位のタイミングなど）は無視されますが、論理設計の検証には十分です。
* **C++ / SystemC への変換（Verilated）:**
    HDL を C++ クラスに変換するため、ソフトウェアエンジニアが使い慣れた C++ テストベンチ（`main.cpp` など）からハードウェアモデルを呼び出すことが可能です。
* **リソース消費が少ない:**
    商用ツールに比べてメモリ消費が極めて少なく、大規模な SoC 設計でも一般的な PC で高速に動作します。

---

## 2. 動作の仕組み：コンパイル・フロー

Verilator は「インタープリタ」ではなく「コンパイラ」として機能します。

1.  **Linter:** 入力された Verilog / SystemVerilog コードの構文チェックを行い、合成不可能な記述や不適切な設計を警告します。
2.  **Elaboration & Optimization:** モジュールの階層を平坦化し、論理式を最適化します。
3.  **Code Generation:** 最適化された論理を **C++ のクラス**（`.h`, `.cpp`）として出力します。
4.  **Compilation:** 出力された C++ ファイルと、ユーザーが書いたテストベンチ（Wrapper）を C++ コンパイラ（GCC や Clang）でビルドし、実行バイナリを作成します。



---

## 3. 得意なこと・苦手なこと

Verilatorは非常に強力なシミュレータですが、完璧というわけではありません。むしろ「尖りすぎていて、かなり割り切ったデメリットや制約（クセ）があるツール」です。
VerilatorがVivadoなどの純正シミュレータと比べて「爆速」なのは、魔法を使っているからではなく、「多くの高度な機能や正確性を大胆に削ぎ落としているから」です。

### 得意なこと（メリット）
* **CPU / プロセッサの開発:** 数千万サイクル回す必要がある CPU のシミュレーション（OS のブート確認など）には最適です。
* **ソフトウェア・ハードウェア協調検証:**
    C++ で書かれたソフトウェア・スタックと RTL モデルを直接リンクできるため、ドライバ開発などがスムーズに進みます。
* **CI/CD との相性:**
    ライセンスサーバが不要で、軽量・高速なため、GitHub Actions などの自動テスト環境に組み込むのが容易です。

### 苦手なこと（デメリット・4つの限界）

FPGAエンジニアやFWエンジニアが現場で直面する、Verilatorの「完璧ではない4つの大罪（限界）」は以下の通りです。

#### 1. 2値（0と1）しか愛せない（4値シミュレーションの非対応）
通常のハードウェアシミュレータ（VivadoやModelSimなど）は、信号の状態を「`0`（Low）」「`1`（High）」「`X`（不定・初期化されていない状態）」「`Z`（ハイインピーダンス・フローティング状態）」の**4つの値**で厳密に計算します。

* **初期化漏れ（`X`）が見つからない:** 電源を入れた直後に値が決まっていないバグ（`X`）があっても、Verilatorは勝手にそれを `0` として処理して何食わぬ顔で動いてしまいます（実機に持っていくと暴走します）。
* **三状態バッファ（`Z`）が通らない:** 双方向ピンの `Z` 状態を理解できないため、基板上のI2Cバスや外部メモリとのやり取りのような「回路の解放」を伴うRTLを書くと、コンパイルエラーではじかれます（F-BBがわざわざ信号線を3本に分解するラッパーを強要しているのは、このVerilatorの致命的な弱点を隠すためです）。

#### 2. 時間の概念がない（遅延記述 `#10` などの無視）
Verilogでは、`#10 out = in;` のように「10ナノ秒後に信号を変化させる」という遅延記述（Gate-level delay）が書けますが、Verilatorはこれを完全に無視（ゼロ遅延として処理）します。
Verilatorは時間経過で回路を追うのではなく、C++の関数のように「クロックが立ち上がったら、一瞬ですべてのロジックを計算し尽くす」という動きをするため、物理的な配線遅延やタイミングのシミュレーション（ゲートレベルシミュレーション）は一切できません。

#### 3. イベント駆動ではなく「全件スキャン」（非効率な回路もある）
普通のシミュレータは、信号が変化した場所（イベント）だけを追いかけて計算します。
しかし、Verilatorはクロックが動くたびに、**回路全体の評価（`eval()`）をC++のコードとして端から端まで力技で実行**します。
そのため、非同期デザイン（あちこちでバラバラのクロックが動いている、あるいはクロックを使わない組合せ回路の塊など）を突っ込むと、C++への変換効率が劇的に落ち、最悪の場合は「Vivadoのシミュレータより遅くなる」か、あるいは「正しく波形が再現できない」という事態に陥ります。

#### 4. 最新のSystemVerilog（UVM）のオブジェクト指向が全滅
SystemVerilogは検証のためにオブジェクト指向などを備えていますが、Verilatorは**「クラス（`class`）」「継承」「動的メモリ確保」といった検証用の高度な文法をほぼサポートしていません。**
Verilatorが解釈できるのは、あくまで「最終的に半導体チップ（シリコン）の形に合成できる、シンセサイザブル（論理合成可能）なRTL記述」だけです。そのため、大手の半導体メーカーが組むような「UVMベースのSystemVerilogテストベンチ」を流し込んでも、1行目でコンパイルエラーを吐いて沈黙します。

* **その他の制限:**
  * **VHDL 非対応:** 基本的には Verilog / SystemVerilog 専用です（VHDLを使う場合は他のツールが必要です）。

## 4. 活用シーン

Verilator は、現代のオープンソース・ハードウェア・ムーブメントの柱となっています。

* **RISC-V 開発:** ほとんどのオープンソース RISC-V プロセッサ（Rocket Chip, LowRISC など）は、検証の主力として Verilator を採用しています。
* **エミュレータ開発:** レトロコンピュータの CPU やカスタムチップの挙動を精密に再現する際、RTL から直接 C++ モデルを生成する用途で使われます。
* **大規模 SoC の論理検証:** 商用ツールのライセンス数が足りない場合、大量の回帰テスト（Regression Test）を Verilator で並列実行させる手法が一般的です。

## まとめ

Verilator は、**「ハードウェアの設計を、ソフトウェアのスピードと柔軟性で検証する」** ための強力なツールです。特に本プロジェクトのような CPU エミュレーションや、自作 CPU の開発、組込みソフトウェアの先行開発においては、これ以上ない選択肢と言えます。
