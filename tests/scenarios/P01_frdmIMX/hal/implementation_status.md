# GPU画像処理HAL実装ステータス (GPU Image Processing HAL Implementation Status)

## 概要
現在、`IVideoProcessor` インタフェースの実装である `ImxGlesProcessorBase` (およびその派生クラスである `Imx8mpGlesProcessor`, `Imx95GlesProcessor`) は、**実機ハードウェア（Production）パスがダミー（モック）実装**となっています。
一方、シミュレーション（ホスト）環境用の `HostGlesProcessor` には、GLSLシェーダを用いた本物の歪み補正・合成（スティッチング）のロジックが実装されています。

| `Imx95GlesProcessor` | i.MX 95 実機 | **ダミー実装 (Mock/Dummy)** | 同上。 |

## 3. シミュレーション（ホスト）ビルドを維持する解決策（モック・アーキテクチャ）

実機SDKやヘッダーをF-BBリポジトリにコミットしたり、ホスト環境のビルドを破壊したりすることなく実機コードを書き進めるため、以下の設計を導入しました。

### ① `__has_include` によるヘッダーの自動判定
C++17の `#if __has_include(<gbm.h>)` 等を使用し、コンパイル環境に `<gbm.h>` などの実機用ヘッダーがあるか自動判定します。ホスト環境などのヘッダーが存在しない環境では、コンパイルを通すための最小限のダミー構造体・定数が自動定義されます。

### ② GBM / DRM 関数の動的ロード（`dlopen` / `dlsym`）
ビルド時に `libgbm.so` への依存（`-lgbm` による静的リンク）を無くすため、実機機能に関連する API は実行時に `dlopen("libgbm.so")` を用いて動的にシンボルをロードします。ホスト環境ではライブラリが存在しないため自動的にダミーフォールバックモードで実行され、実機環境ではロードが成功して本物のパスが起動します。

### ③ 実機における同期保証（`glFinish` によるGPU/CPU同期）
実機ボード上のGPU（Vivante / Mali）特有の不確定要素として、GPUの描画完了を待たずに `glReadPixels` が実行されてしまい、ピクセルデータの読み出しに時間的なズレ（同期ラグ）やキャッシュコヒーレンシの不整合が発生することがあります。これを防ぐため、`glReadPixels` の直前で明示的に `glFinish()` を呼び出し、描画が100%完了するまでCPUスレッドを確実にブロックさせてから読み戻すように設計しています。

## 4. 今後の実装計画 (ロードマップ)

実機へのデプロイおよび動作確認を行うフェーズにおいて、以下の手順で Production パスを実装します。

### Step 1: 実機上での GBM / EGLImage パイプラインの検証
今回構築したモックコンパイル用プレースホルダーの `processFrame` 内で、DMA-BUF fd の確保および `eglCreateImageKHR` へのマッピング処理の記述を本番用にブラッシュアップします。

### Step 2: 実機SDK環境への移行とビルド
NXPのYoctoベースSDK（あるいは対応する実機クロスコンパイル環境）でビルドを実行します。実機上では自動的に本物のヘッダーが検出され、動的ロードが有効化されます。

### Step 3: ゼロコピー描画の同期制御デバッグ
CPU/GPU間のアクセスレースを防ぐためのフェンス制御や、メモリキャッシュの一貫性（キャッシュフラッシュ）処理を追加し、実機ハードウェア上での安定動作を検証します。
