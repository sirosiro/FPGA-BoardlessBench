# tests の使い方

## 概要

tests/ には、DTS と Verilog、Cコードが含まれており、それらを組み合わせることで、FPGAが提供するデバイスを操作するFWを作成してテストすることができます。

## 各ファイルの説明

1. **DTS (`config.dts`):**
   - 「どんなデバイスが、どの番地に、どんな名前であるか」を定義する **唯一の仕様書**。
2. **Verilog (`vfpga_top.v`):**
   - 「その番地を叩いた時に、ハードウェアがどう動くか」を記述。
   - **118ピン標準インターフェース** (`l_pins_i/o/t [117:0]`) を用いて記述します。
   - 複数の `.v` ファイルに分割して配置することも可能です。
3. **Cコード (`main.c`):**
   - 「FWからデバイスをどう操作するか」を記述。
4. **Makefile:**
   - シナリオごとのビルド（FWのコンパイル設定等）を定義。実機への移植性を高めるため、`CC ?= gcc` の形式で記述します。

基本的には、DTS と Verilog を用意し、その後で C コードを作成して動作確認を行います。

> [!NOTE]
> **【実機開発との違いについて（重要）】**
> 実際のFPGA開発（Vivado等）では、「FPGAのハードウェア回路（Verilogコードなど）を作ってから、それに合わせてDTSが自動生成される」という流れになります。
> しかし FPGA-BoardlessBench (F-BB) ではエミュレータの構造上、**「DTS を唯一の仕様書として書き、そこから Verilog のバス配線やポート宣言（スケルトン）が自動生成される」** という逆の流れを採用しています。学習の際は、この逆転の仕組みを頭に入れておいてください。

## シナリオ追加時のルール（完全隔離とミニマリズム）

`tests/scenarios/` 配下に新しいテスト（シナリオ）を追加する際は、以下のルールを守ってください：

- **完全な独立性:** 各シナリオディレクトリは独立した「1つのプロジェクト」として完結させてください。他のシナリオのDTSや状態に依存するような構成にしてはいけません。
- **マルチソース対応:** 複数の Verilog ファイル (`*.v`) を配置した場合、Verilator によって自動的に統合・ビルドされます。これによりモジュール化された回路設計のテストが可能です。
- **ミニマリズム（ノイズの排除）:** そのシナリオでテストしたい機能に関係のない無駄なペリフェラル定義をDTSに残さないでください。コードを読む人が「そのディレクトリのファイルだけを見れば全てが理解できる」ピュアな状態を保つことが重要です。

## FWエンジニアのための F-BB 適合（ポーティング）ガイド

物理ボードや完成したRTLがない開発初期段階において、FWエンジニアが自身のプロジェクトを F-BB に適合させ、先行開発を行うための実践ガイドです。

### 1. DTS の F-BB 用カスタマイズ（味付け）
FPGAエンジニアや Vivado から出力された標準のデバイスツリー（DTS）は、そのままでは F-BB のダッシュボードやシミュレータと連動できません。以下の **3点の手修正** を加えて `config.dts` として配置します。

#### 📝 カスタマイズの具体例
```devicetree
/* FPGAエンジニアから貰った標準DTS */
my_device@42000000 {
    compatible = "vfpga,custom-ip-1.0";
    reg = <0x42000000 0x1000>;
};

【F-BB用にデコレーション】

my_device@42000000 {
    compatible = "generic-uio";   /* ① compatible を "generic-uio" に変更 (Shimでのフック用) */
    reg = <0x42000000 0x1000>;
    label = "/dev/uio0";          /* ② FWアプリケーションがオープンするデバイス名を指定 */
    
    /* ③ レジスタマップ仕様書を元に、名前とオフセットを追記 (ダッシュボード表示 & シミュレータ同期用) */
    registers = "CTRL @ 0x00, SPEED @ 0x04, STATUS @ 0x08"; 
};
```
*   **`registers` プロパティの書式**: `"レジスタ名 @ オフセット, レジスタ名 @ オフセット"` の形式でカンマ区切りで記述します。ここに追加した名前がそのままダッシュボードの履歴グラフやテーブルに反映されます。

### 2. 最上位モジュール `vfpga_top.v` の「強制ルール」

F-BBのシミュレータ環境でシミュレーションを実行する場合、最上位の Verilog ファイルとモジュールには **一切の変更・カスタマイズが許されない「プラットフォーム上の制約（強制ルール）」** が存在します。

> [!IMPORTANT]
> **強制ルール 1: 最上位ファイル名とモジュール名の固定**
> *   最上位ファイル名は必ず **`vfpga_top.v`** でなければなりません。
> *   最上位モジュール名は必ず **`module vfpga_top`** でなければなりません。
> *   **理由**: F-BBの C++ シミュレータ・コア（`src/sim/sim_main.cpp`）およびビルドエンジンが、Verilator によって自動生成される `Vvfpga_top.h` および `Vvfpga_top` クラスと静的にハードコードで結合しているためです。この名前を変更するとコンパイルが通りません。

> [!IMPORTANT]
> **強制ルール 2: 仮想バス（抽象化レイヤー）とGPIOポート定義の準拠**
> 最上位の `vfpga_top` モジュールは、シミュレータが共有メモリの物理アドレス空間と双方向同期を行うために、必ず以下のポート（ピン）構成を備えていなければなりません。

```verilog
module vfpga_top (
    input wire clk,              // システムクロック
    input wire rst_n,            // アクティブLowリセット

    // ── F-BB 仮想バスインターフェース (抽象化レイヤー) ──
    input wire [31:0] addr,      // 共有メモリを叩いた「32bit物理絶対アドレス」が直接供給される
    input wire [31:0] w_data,    // 書き込みデータ
    input wire w_en,             // 書き込み有効化 (1のとき書き込み)
    output reg [31:0] r_data,    // 読み出しデータ

    // ── ダッシュボード UI 連動用 GPIO ピン (SoC標準 118ピン) ──
    input wire [117:0] l_pins_i, // ダッシュボード UI（トグルスイッチ）からの入力信号
    output wire [117:0] l_pins_o,// ダッシュボード UI（LEDインジケータ）への出力信号
    output wire [117:0] l_pins_t  // 各ピンの方向制御 (0:出力としてLED連動, 1:入力としてトグル連動)
);
```

#### 仮想バス（抽象化レイヤー）の動作メカニズム
実機の `AXI-Lite` などの複雑なバスプロトコルを極限まで単純化し、以下のように動作します。
*   **書き込み時**: FWが共有メモリへ書き込みを行うと、シミュレータがそれを検知し、`addr` にアドレス、`w_data` にデータを乗せて `w_en = 1` にします。RTLはこのタイミングで内部レジスタに値を取り込みます。
*   **読み出し時**: シミュレータは常時 `addr` バスに読み出したいアドレスを流しています。RTLは `addr` に応じたレジスタ値を `r_data` に出力し続け（組み合わせ回路）、シミュレータがそれを共有メモリへ自動的に書き戻します。

### 3. 実機RTLを汚さない「ラッパー戦略」
FPGAエンジニアが設計した本番用のIP（例: AXI-Liteバス搭載の回路）のコードを、F-BB向けに修正する必要は一切ありません。
上記の強制ルールを満たす `vfpga_top.v` を **「ラッパー（中継器）」** として作成し、その内部で本番用IPを呼び出す（インスタンス化する）構造にします。

```verilog
// ── vfpga_top.v (F-BB用の繋ぎ込みラッパー) ──
module vfpga_top ( ... ); // 強制ルールに準拠したポート定義

    // F-BBのシンプルな仮想バスを、本番IP用の AXI-Lite バスへ中継・配線する
    my_actual_production_ip u_ip (
        .clk      (clk),
        .rst_n    (rst_n),
        .axi_addr (addr),     // F-BBの物理アドレスをそのまま配線
        .axi_wdata(w_data),    // F-BBのデータを配線
        .axi_wen  (w_en),
        .axi_rdata(r_data)
    );

    // 118ピンGPIOの方向と出力を配線
    assign l_pins_o = {86'h0, u_ip.led_output}; // 下位32ビットに本番のLED出力をマップ
    assign l_pins_t = 118'hFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF; // 全て入力(1)または適宜TRIを配線

endmodule
```
This wrapper design ensures that the **actual hardware IP core RTL (`my_actual_production_ip.v`) remains completely untouched** and devoid of any simulation-specific code. This maintains a clean and seamless bridge between physical deployment and boardless simulation.

### 4. 開発初期段階（モック開発）の 3 ステップ

1.  **DTSだけでレジスタ定義を決定**:
    合意したレジスタ仕様書を元に `config.dts` のみを記述して配置します。
2.  **RTLスケルトン（空モック）の自動生成**:
    まだ `.v` ファイルを用意していない状態で `./tests/run_tests.sh` を叩くと、F-BBのジェネレータが **DTSと完全に同期した `src/rtl/vfpga_top.v` のレジスタ読み書きボイラープレート（空モック）を自動生成** してくれます。
    *   **ローカル優先ルール（カスタマイズ方法）**: 独自のRTL（`pattern_engine.v` など）を配線して検証したい場合は、自動生成された `src/rtl/vfpga_top.v` を `tests/scenarios/<scenario_name>/vfpga_top.v` にコピーして配置し、そこを編集してください。**F-BBのビルドエンジンは、シナリオフォルダ配下に置かれた Verilog ファイルを最優先でコンパイルに使用し、自動生成された共通ファイルを自動的にビルド対象から外します**。
3.  **擬似ロジックの肉付け**:
    自動生成されたスケルトンの内部レジスタに、FW開発に必要な簡易的なモック動作（例: CTRLレジスタに特定のビットが立ったらSTATUSレジスタのビットを立てる、カウンタをインクリメントするなど）を数行記述し、FWの対話型デバッグを開始します。

### 5. 実機環境への移植コスト（オーバーヘッド）と注意点

F-BB上で先行開発・動作確認を完了したソフトウェア（FW）とハードウェア（RTL）を、最終的に実機ターゲットボードへ移植してビルドする際、**「シミュレーション用の記述を削除する」「本物の信号と辻褄を合わせる」といったコードレベルの同期・修正の手間は 1ミリも発生しません（工数ゼロ）。**

F-BBのアーキテクチャが「完全に非侵襲的（Non-intrusive: 相手のコードを汚さない）」に設計されているため、以下の理由でそのまま実機ビルドへ移行できます。

#### 実機移植時の手間が「ゼロ」である3つの技術的理由

1.  **FW（Cコード）の透過性**:
    F-BB上での動作時、FWは標準の `/dev/uio0` などを `open/mmap` しており、F-BBはそれを実行時に `LD_PRELOAD` ライブラリ（`libfpgashim.so`）で動的に横取りしているだけです。
    実機では単に `LD_PRELOAD` なしで起動するだけで、**FWのCソースコードを1行も変更することなく、本物の物理アドレス空間（実ハードウェア）へ透過的にアクセス**します。
2.  **RTL（Verilog）の完全なクリーン化**:
    本番用のIPコア（`pattern_engine.v` など）には、F-BB独自のバスインターフェースやデバッグ用ポートは一切含まれていません。「ラッパー（`vfpga_top.v`）戦略」によって隔離されているため、**本番IPのRTLソースコード自体を実機ビルド用に修正する手間は完全にゼロ**です。
3.  **DTS（デバイスツリー）の互換性**:
    F-BB用に追記したカスタム属性（`registers = ...` など）は、実機Linuxのデバイスツリーコンパイラ（DTC）やLinuxカーネルによって**「単に無視される未知のカスタムプロパティ」として安全にスルー**されます。DTSの修正をしなくても実機Linuxシステム上でエラーなく動作します。

#### 実機ビルド環境での注意点（ビルド時の除外）

実機ビルド（VivadoやPetaLinuxなどの実機開発環境）を実行する際、唯一必要となる手作業は以下の通りです。

> [!WARNING]
> *   **RTLビルド時**: 実機プロジェクト（Vivado等）にコードを取り込む際、最上位中継ファイル **`vfpga_top.v` を「シミュレーション専用（Simulation Only）」に設定するか、プロジェクトから除外（Remove）** してください。代わりに、本番用IP（`pattern_engine.v` 等）を、実機向けのブロックデザイン（IP Integrator）や実機用トップモジュール内で、**本物の `AXI GPIO` ブロックや `AXI-Lite` バスへ配線（マウスによる接続やコード記述）** します。
> *   **FWビルド時**: F-BBの Shimライブラリ（`libfpgashim.so`）や、ダッシュボード側のNode.jsプログラムは実機には配布しません。

この簡単な除外・配線設定（Vivado等のGUI上でのチェック切り替えと配線）さえ行えば、F-BBで磨き上げたFWとRTLはそのまま実機ボード上で完璧に動作するはずです。その後はいよいよ本番のFPGA開発環境でのデバッグを始められます。健闘を祈ります！
