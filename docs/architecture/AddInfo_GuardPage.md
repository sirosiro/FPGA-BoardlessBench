# Guard Page（ガードページ）のハードウェア・エミュレーション

C-Shim（C言語で書かれたハードウェア・エミュレーション用のシムレイヤ）などにおいて、「共有メモリ（レジスタ空間）のガードページ化」**という手法が使われることがあります。これは、実機レス（ハードウェアなし）の開発環境やエミュレータにおいて、**「ファームウェアが特定のレジスタ（メモリ空間）にアクセスした瞬間を、OSのメモリ保護機能を使ってトラップ（捕捉）する」ためのテクニックです。

ASanがコンパイル時にチェックコードを「埋め込む」のに対して、こちらは**OSのページテーブル（MMU）の仕組みをそのまま利用する**ため、ファームウェア側のコードに一切手を加えることなく、特定のメモリ読み書きをフックできるのが大きなメリットです。

---

## 1. ガードページ化の基本原理

OSはメモリを通常4KBなどの「ページ」単位で管理しており、ページごとに「読み取り専用」「読み書き可能」「アクセス禁止（プロテクト）」といった権限を設定できます。

> **仕組みのイメージ**
> 1. FPGAやSoCの「レジスタ空間（レジスタマップ）」に見立てたメモリ領域を確保する。
> 2. その領域（ページ）のアクセス権限を、OSのAPI（Linuxなら `mprotect()`、Windowsなら `VirtualProtect()`）を使って**「アクセス禁止（`PROT_NONE`）」**に設定する（＝ガードページ化）。
> 
> 

ファームウェア（Cコード）がこの領域のレジスタを読み書きしようとすると、CPU（MMU）がアクセス違反を検知し、OSがプログラムに対してセグメンテーション違反（`SIGSEGV`）などのシグナル（例外）を発行します。

---

## 2. アクセスを検知してエミュレーションを動かす流れ

単にアクセス禁止にするだけだとプログラムがクラッシュして終わってしまいますが、これをシグナルハンドラ（例外ハンドラ）と組み合わせることで、「レジスタの読み書きエミュレーション」へと昇華させます。

1. **シグナルハンドラの登録:** 初期化時.
プログラム起動時に `sigaction()`（Linux）などを使い、`SIGSEGV` が発生したときに自作のハンドラ関数が呼ばれるように設定しておきます。


2. **ファームウェアのレジスタアクセス:** 実行時.
ファームウェアが `*(volatile uint32_t *)(REG_BASE + 0x10) = 0x01;` のように、ガードページ化されたレジスタ空間へアクセスします。


3. **MMUによるトラップと例外発生:** ハードウェア/OS.
CPUのMMUがアクセス不可を検知し、即座に実行を中断して登録しておいたシグナルハンドラへ制御を移します。


4. **ハンドラ内でのエミュレーション実行:** シグナルハンドラ内.
ハンドラに渡される構造体（`siginfo_t` や `ucontext_t`）から、**「どのアドレスに」「どんな値を書き込もうとしたか（または読み出そうとしたか）」**および**「クラッシュした命令のPC（プログラムカウンタ）」**を特定します。そのアドレスに対応する仮想ハードウェア（PythonやCで書かれたシミュレータなど）の挙動を動かします。


5. **プロテクトの一時解除と命令の復帰:** ハンドラ終了時.
対象ページのプロテクトを一時的に解除（`PROT_WRITE` などに）し、失敗した命令を1ステップだけ実行させてから、再度ガードページに戻します。または、PC（プログラムカウンタ）を次の命令へ進めるように書き換えて、ハンドラから通常実行へ復帰させます。


---

## 3. なぜこの手法を使うのか？（メリット）

### ① ファームウェアのソースコードが「実機と全く同じ」で済む

もしこれを行わない場合、レジスタアクセスの部分を `RegWrite(0x10, 0x01)` のようなエミュレータ用の関数マクロに置き換える必要があります。しかしガードページ化を使えば、実機向けのポインタ直接参照（`*(volatile uint32_t *)`）のコードを**1行も書き換えることなく**そのままPC上で動かせます。

### ② ハードウェアとの「非同期・イベント駆動」のシミュレーションが容易

ファームウェアがステータスレジスタをポーリング（監視）し始めた瞬間を確実に捉えられるため、シミュレータ側（別スレッドやPython側など）へ「今、ポーリングが始まったからステータスを更新して割り込みを入れろ」といったイベントを遅延なく通知できます。

---

## 4. デメリットと注意点

* **オーバーヘッドが非常に大きい**
メモリバグを高速に検知するASanとは対照的に、この手法はレジスタアクセス1回ごとに「OSのコンテキストスイッチ（ユーザー空間 ↔ カーネル空間の往復）」とシグナル処理が発生します。そのため、ミリ秒単位で数万回もポーリングを行うようなループをそのまま回すと、実行速度が劇的に低下します。
* **ページ単位（通常4KB）での大雑把な管理**
OSのメモリ保護はバイト単位ではなく「ページ単位」です。そのため、4KBの同じページ内に「頻繁にアクセスする通常の変数」と「監視したいレジスタ空間」が混在していると、関係のないアクセスでも毎回トラップが発生してしまい、処理が破綻します。レジスタ空間は必ず `posix_memalign()` などでページ境界にアライメント（整列）して確保する必要があります。
