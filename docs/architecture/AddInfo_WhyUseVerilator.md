# それでもなぜVerilatorが使われるのか？（F-BBの割り切り）

Verilatorにはいくつかの強力な制限や弱点（4値非対応、遅延記述の無視、全件スキャンによる非効率性、UVM非対応など）があります。
これほど欠点だらけなのに、なぜF-BBの核として選ばれたのか。理由はシンプルで、VerilatorとF-BBの「割り切り方（物理的・時間的な厳密さを削ぎ落とし、ソフトウェアとしての動作速度と検証の容易さをとる）」という基本思想が完全に一致しており、抜群に相性が良いからです。

F-BBが目指すものは、タイミング（遅延や物理的制約）の検証ではなく、純粋な **「論理検証（ロジック検証）」** です。そして、それを **「ファームウェアとハードウェアの協調論理検証（Co-verification）」** に適用することにより、Verilatorが抱える4つの弱点は以下のようにほぼすべて無害化（あるいはメリットに転化）されます。

### 1. 「4値非対応（X/Z）」が問題にならない理由
* **不定状態 `X` の無害化:** F-BBのシミュレーション環境では、電源投入直後にシミュレータ側（`sim_main.cpp`）で必ず論理的なリセット系列を実行し、すべてのレジスタや内部状態を一度クリーンに初期化します。そのため、実際のハードウェアで起こるような「電源ON直後の不定値による暴走」のデバッグは不要であり、`X` の非対応は実用上問題になりません。
* **ハイインピーダンス `Z` の回避:** そもそもFPGAの内部配線には `Z`（三状態バッファ）は存在せず、すべて `0` と `1` で論理を組みます（`Z` が現れるのは外部ピンのみ）。F-BBでは、双方向ピン（GPIOやI2Cなど）をRTLレベルで「入力信号」「出力信号」「出力イネーブル信号」の3本の単方向信号に分解してラッピングする設計になっているため、シミュレータが `Z` を解釈できなくても全く問題ありません。

### 2. 「遅延記述（#10など）の無視」が問題にならない理由
FW（C++）とRTLモデルの協調動作において重要なのは、ナノ秒単位のタイミング（セットアップ・ホールド時間など）ではなく、「レジスタへ書き込み、次のクロックで値が反映され、ステータスレジスタを正しく読み出せるか」という**クロックサイクル単位での論理同期（RTL動作）** だけです。
論理的なサイクル精度さえ正確であれば、物理遅延がゼロであってもFWの動作検証には全く影響しません。物理タイミングの検証は配置配線ツール（Vivado等）に任せ、FW開発フェーズでは「遅延無視＝超高速シミュレーション」のメリットだけを享受しています。

### 3. 「全件スキャンによる非効率性」が問題にならない理由
F-BBで検証対象とするのは、巨大なSoC全体ではなく、ターゲットとなる「周辺IPコア単体」や「シンプルなペリフェラル」のみです。
扱う回路規模が小さく、かつ単一（あるいは極めて少数）の同期クロックで設計されているため、たとえ毎クロック全回路を愚直に走査（全件スキャン）したとしても、PC側で処理する計算負荷は非常に軽微です。結果として、イベント駆動型の商用シミュレータよりもはるかに高速に動作します。

### 4. 「SystemVerilog UVM非対応」が問題にならない理由
F-BBの検証アプローチは、FWエンジニア自身がC++（`main.cpp`）やPythonを使って、テストプログラムそのものを書くスタイルです。
半導体検証チームが使うような、SystemVerilog UVMなどの複雑なオブジェクト指向テストベンチを組む必要はありません。むしろ、C++に直接コンパイルされるVerilatorのおかげで、FWエンジニアは**使い慣れたC++でテストシナリオをガシガシ記述できる**という抜群の親和性が生まれています。

---

「完璧ではない。だが、特定の用途（FWとの協調）においては他のどのツールよりも圧倒的に高速で扱いやすい」。このVerilatorの割り切った特性を理解した上で、その限界をシステムのガワ（DTSやラッパー、シナリオ）で覆い隠して実用的なボードレス開発環境に仕立て上げたF-BBは、非常に割り切りの利いた合理的な設計をしています。
