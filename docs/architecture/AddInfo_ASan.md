# AddressSanitizer（ASan） 組み込みデバイスのメモリ安全性強化

**AddressSanitizer（通称 [ASan](https://clang.llvm.org/docs/AddressSanitizer.html)）**は、Googleが開発したC/C++プログラム向けの**メモリ誤り検出ツール**です。

コンパイラ（GCCやClangなど）にビルドオプションとして組み込まれており、実行時の深刻なメモリバグを高速かつ高精度にキャッチしてくれます。従来のツール（Valgrindなど）に比べて動作が圧倒的に軽い（実行速度の低下が2倍程度に収まる）ため、開発時のデバッグやCI/CD（自動テスト）のプロセスに組み込みやすいのが大きな特徴です。

---

### 主に検出できるバグ

ASanは、主に以下のような「不正なメモリ領域へのアクセス」を監視します。

* **Out-of-bounds（境界外アクセス）:**
配列の要素数を超えて読み書きしてしまうバグ。ヒープ、スタック、グローバル領域のすべてに対応。
* **Use-after-free（解放後参照 / ダングリングポインタ）:**
`free()` や `delete` で解放済みのメモリ領域に、その後もアクセスしてしまうバグ。
* **Use-after-return（関数を抜けた後のスタック参照）:**
ローカル変数のポインタを関数の外に持ち出してしまい、関数終了後にアクセスするバグ（オプションで有効化）。
* **Double free / Invalid free:**
同じメモリを2回解放したり、確保していないポインタを解放したりするミス。

---

### 仕組み：シャドウメモリ（Shadow Memory）

ASanの高速な検出を支えているのが「シャドウメモリ」というアーキテクチャです。

> **仕組みのイメージ**
> アプリケーションが使う実際のメモリ（メインメモリ）の全領域に対して、その場所が「今アクセスして安全かどうか」を記録した**メタデータ用の地図（シャドウメモリ）**を、メモリ全体の「8分の1」のサイズで裏側に確保します。

1. **コンパイル時のコード埋め込み:**
コンパイラが、メモリへのアクセス（ロード/ストア）が発生するすべてのコードの直前に、「シャドウメモリをチェックするコード」を自動的に挿入します。
2. **毒化（Poisoning）:**
アクセスしてはいけない領域（配列の隙間や、解放された領域）に対応するシャドウメモリに「毒（Poison）」のマークをつけます。
3. **実行時のチェック:**
プログラムがメモリにアクセスした瞬間、その場所が「毒化」されていないかを高速にチェックし、もし毒に触れていたらその場でプログラムを強制終了（アボート）して、詳細なエラーレポート（クラッシュ時のコールスタックなど）を吐き出します。

---

### 使い方（GCC / Clang）

使い方は非常にシンプルで、コンパイル時とリンク時のオプションに `-fsanitize=address` を追加するだけです。デバッグ情報を残すために `-g` も併用するのが一般的です。

```bash
# コンパイルとリンク
gcc -fsanitize=address -g my_program.c -o my_program

# 実行（バグがあれば、この時点で詳細なレポートが表示されて停止します）
./my_program

```

メモリリークも同時に検出したい場合は、環境変数で `ASAN_OPTIONS=detect_leaks=1` を指定して実行することで、派生ツールである **LeakSanitizer（LSan）** の機能も連動させることができます（Linux環境などではデフォルトで有効）。
