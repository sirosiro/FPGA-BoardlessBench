# FPGA-BoardlessBench (BB) における RTOS / ベアメタル FW のエミュレーション構想

FPGA-BoardlessBench (BB) は本来、Linux上のユーザー空間アプリケーション（Linux FW）が `/dev/mem` や `/dev/uio0` を介して FPGA にアクセスする挙動を `LD_PRELOAD` でインターセプトすることでシミュレーションを実現しています。

しかし、近年主流の **AMP（非対称マルチプロセッシング：LinuxとRTOSの協調動作）** アーキテクチャを完全にシミュレーションするには、Linux FWだけでなく **「RTOS やベアメタルで動く FW」** もBB環境上でエミュレートする必要があります。本ドキュメントでは、その実現に向けた技術的なアプローチと既存の優秀なライブラリを用いた設計方針をまとめます。

---

## 1. 課題：なぜ RTOS FW はそのままでは動かないのか？

RTOS やベアメタルの FW は、Linux OS の管理下にないため、`open` や `mmap` といった Linux のシステムコールを使用しません。
代わりに、以下のような方法で FPGA のレジスタ（物理メモリ）にアクセスします。

1. **直接ポインタアクセス**: `*(volatile uint32_t*)(0x40000000) = 1;`
2. **ベンダー提供のHAL（Hardware Abstraction Layer）**: Xilinx環境における `Xil_Out32(0x40000000, 1);` など

これを BB 環境（ホストPCの Linux OS 上）でそのままコンパイルして実行しようとすると、ユーザー空間から無効な物理メモリアドレスへ直接書き込もうとするため、即座に **Segmentation Fault（セグフォ）** が発生してクラッシュしてしまいます。

したがって、Linux FW のように「実行時にシステムコールをフックする (`LD_PRELOAD`)」という手法は使えません。

---

## 2. 解決策：システムコールフックから「HAL/OSのモック化」へ

RTOS / ベアメタル FW の**メインロジック（業務仕様）を一切書き換えることなく** BB 環境で動かすためには、実行時ではなく **「ビルド時に下回りのインターフェースを差し替える（モック化する）」** アプローチを採用します。

### 2.1. レジスタアクセス（HAL）のモック化

末端のメモリアクセス処理を、BB の仮想 FPGA（共有メモリ）へのアクセスにすげ替えます。

* **ベンダーHALを使用している場合**
  FW が `Xil_Out32(addr, val)` や `Xil_In32(addr)` などを呼び出している場合、ホスト PC 向けにコンパイルする際、実機用ライブラリの代わりに **「BB用のダミー HAL ライブラリ」** をリンクさせます。
  ダミー HAL 内で `Xil_Out32` を自作し、その内部で BB の共有メモリ（POSIX Shared Memory）の該当オフセットにデータを書き込むように実装します。
  
* **マクロによる直接アクセスをしている場合**
  FW 内に `#define REG_A (*(volatile uint32_t*)(0x40000000))` のような定義がある場合、BB 環境用のビルドフラグ（例: `#ifdef SIMULATION`）を用いて、
  `#define REG_A (bb_hal_read32(0x40000000))` のように、共有メモリアクセス関数へ置換するマクロを注入します。

### 2.2. RTOS 特有の API（OS層）のモック化

FW が `xTaskCreate`（タスク生成）や `xQueueSend`（キュー送信）など、FreeRTOS 等の専用 API を使っている場合、これもホスト PC（Linux）上ではそのまま動きません。

これを解決するために、F-BB側でOS抽象化レイヤー（OSAL）を独自に開発してモック化する必要はありません。すでにオープンソースコミュニティで公開されている**「POSIX（Pthreads/シグナル）上でRTOSの挙動を再現する移植レイヤー・ラッパーライブラリ」**を利用することで、工数をかけずかつ堅牢にOSAPIの置き換えが可能です。

### 2.3. F-BBの責務とテストシナリオにおけるライブラリ管理方針

既存のRTOSラッパーライブラリをF-BBに統合するにあたり、リポジトリの設計・運用上の役割分担（責務の分離）を以下のように定義します。

* **F-BBコアの提供範囲（責務）**:
  F-BBは、スレッドもしくはプロセスによってマルチコアを表現し、それぞれでLinuxが動いている環境（共有メモリを介した仮想FPGAへの同時アクセス等）を提供することに専念します。F-BBの共通コアリポジトリ内に特定のRTOSラッパーライブラリを同梱・管理することはしません。
* **ユーザーによる自前選定**:
  ユーザーは、シミュレート対象とするSoC環境やターゲットOS（FreeRTOS、μITRON、CMSIS等）の構成に合わせて、最適なRTOSラッパーライブラリを自前で調査・選定します。
* **テストシナリオ配下での管理・ビルド**:
  選定したRTOSラッパーライブラリは、F-BBの共通領域ではなく、個別の**テストシナリオ配下（例：各シナリオの `hal/` ディレクトリ等）**に格納します。そこで、実機用のオリジナルHALコードやBB用のモックHALと一括してバージョン管理を行い、シナリオ個別のMakefile等でビルドする方針とします。

### 2.4. コード透過性100%を実現する自動mmapインジェクション（実装済み）

物理ポインタを直接操作する実機コードを一切変更せず、ホストLinuxのセグメンテーション違反（セグフォ）を回避するアーキテクチャとして、**「Remoteproc駆動による自動mmapインジェクション」** が実装されています。

* **課題の根本解決**:
  実機用のMコアコード内の `*(volatile uint32_t*)(0x40001000) = 1` といった物理アドレスへの直接アクセスを、シミュレーション側のビルド時マクロや仮想HALに書き換えることなく、100%透過的に実行させます。
* **トリガー同期による起動**:
  Aコア側のアプリケーション（または起動スクリプト）が、実機のリモートコア制御（`Remoteproc`）の Sysfs インターフェース（例: `echo start > state`）を模擬した仮想Sysfsに書き込んだことをフック（C Shimによるオープンリダイレクト）し、Pythonのバックエンドコントローラ（`vlogic_controller` 内の監視スレッド）が検知して、Mコア用プロセスをバックグラウンドで起動します。
* **`MAP_FIXED` による実物理アドレスのマッピング**:
  Mコアプロセスの起動時、ローダーによって自動的に環境変数 `FBB_MCORE=1` が設定され、Mコアプロセス（バイナリ）内の `main` 関数が呼ばれる前（`__attribute__((constructor))` スタートアップコンストラクタ）に、ホストLinux of 仮想アドレス空間として**実機と同じ物理アドレス（例: `0x40001000`）へ `MAP_FIXED` を用いてF-BBの共有メモリを強制マッピング**します。
* **得られる効果**:
  これにより、Mコアのソースコードを1文字も修正することなく（透過性100%）、ホストLinux上でセグフォを起こさずに直接物理アドレス宛てにレジスタアクセスを実行させ、Verilatorシミュレータにルーティングできます。

詳細なアーキテクチャ定義やシーケンスについては、[AddInfo_RTOS_RemoteprocAmap.md](AddInfo_RTOS_RemoteprocAmap.md) を参照してください。

## 3. 推奨されるPOSIX上RTOSラッパー・ライブラリ (既存OSSの活用)

検証環境への組み込みやすさを考慮し、依存関係が極めて少なく、法的リスクやライセンス管理ポリシー上も安全な**MITライセンス（またはそれに準ずる極めて寛容なライセンス）**の軽量C/C++ライブラリを選定しています。

### 3.1. FreeRTOS POSIX Simulator Port (FreeRTOS公式)

FreeRTOSのシステムコールをLinuxのPOSIXスレッドやシグナルにマッピングして、ホストPC上でタスクスケジューリングやキューを完全エミュレートする公式の移植レイヤーです。

* **GitHubリポジトリ**: [FreeRTOS/FreeRTOS-Kernel](https://github.com/FreeRTOS/FreeRTOS-Kernel)
* **シミュレータコードの格納先**: `portable/ThirdParty/GCC/Posix/`
* **ライセンス**: **MIT License**
* **特徴**:
  FreeRTOS公式が直接メンテナンスしているため信頼性が非常に高いです。実機ファームウェアのFreeRTOS依存コード（`xTaskCreate`や`xQueueSend`など）を1行も書き換えることなく、Linux上のネイティブなC++シミュレータコンポーネントとしてリンク・実行させることができます。

### 3.2. micro-ITRON API Wrapper on POSIX

μITRON 4.0のシステムコール（`cre_tsk`, `wai_sem`, `snd_dtq`など）を、POSIXスレッド上にマッピングするための軽量ラッパーライブラリです。

* **GitHubリポジトリ**: [asi-mo/uitron-wrapper-on-posix](https://github.com/asi-mo/uitron-wrapper-on-posix)
* **ライセンス**: **MIT License**
* **特徴**:
  RTOSのカーネル自体を移植するのではなく、システムコール呼び出しをPthreadsの関数に「翻訳」するマクロおよびインライン関数群で構成されています。
  C++のヘッダーと少数のソースだけで完結しているため、コンテナ環境での軽量なビルドやF-BBの個別テストシナリオへの引き込みが極めて容易です。

### 3.3. CMSIS-RTOS2 POSIX Implementation

ARM Cortex-M系で標準的によく使われるRTOS抽象化APIレイヤー「CMSIS-RTOS2 (osCMSIS)」を、Linux/POSIX環境上でエミュレートするためのラッパーです。

* **GitHubリポジトリ**: [nlohmann/cmsis-rtos2-posix](https://github.com/nlohmann/cmsis-rtos2-posix)
* **ライセンス**: **MIT License**
* **特徴**:
  JSONライブラリ（nlohmann/json）の作者であるNiels Lohmann氏らによって開発されたクリーンなC++実装です。実機でFreeRTOSやRTXの上にCMSIS-RTOS2ラッパーを被せている場合、コンパイルターゲットをLinuxに変えるだけでそのままホスト上で駆動させることができます。

### ライセンスと運用のメリット
TOPPERSプロジェクト系（JSPやATK2など）のシミュレータも優秀ですが、それらは「TOPPERSライセンス」という独自の緩いライセンスを採用しています。企業のソース管理ポリシーによっては別途確認が必要になる場合がありますが、上記のように**ストレートなMITライセンス**のライブラリに統一することで、リーガルリスクや依存関係の確認コストを最小限に抑えられます。

## 4. 実現するアーキテクチャと期待される効果

本エミュレーション設計により、テストシナリオ側で以下のようなモック構造をビルド時に構成します。

1. **FW メインロジック**: 実機と全く同じ C/C++ コード
2. **OS 移植・ラッパー層**: ユーザーがシナリオ配下で管理するRTOSラッパー（FreeRTOS POSIX Port等）
3. **HAL モック層**: ユーザーがシナリオ配下で管理するBB用ダミーHAL（メモリアクセスをBB共有メモリへルーティング）

### 期待される効果：AMPアーキテクチャの完全シミュレーション

F-BBは、スレッドもしくはプロセスによってマルチコアを表現し、それぞれでLinuxが動作しているシミュレーション環境（共有メモリを介した仮想FPGAアクセス）を提供します。これにより、以下のプロセスまたはスレッドが協調動作するAMPシミュレーションが実現します。

1. **FPGA シミュレータプロセス** (`vfpga_sim`): バックグラウンドで動作するVerilatorベースのRTLシミュレータ
2. **Linux FW プロセス/スレッド** (コア0相当): `LD_PRELOAD` によるシステムコールフックを利用して仮想FPGAへアクセスするメインのLinuxファームウェア
3. **RTOS/ベアメタル FW プロセス/スレッド** (コア1相当): ユーザーがテストシナリオ配下に用意したラッパーライブラリおよびHALを用い、仮想FPGAの共有メモリへアクセスするRTOSファームウェア

F-BBが提供するマルチプロセス・マルチスレッド実行環境と共有メモリIPCにより、「Linux（コア0相当）とRTOS（コア1相当）が、1つの仮想FPGAへ同時に非同期アクセスする」という **AMP（非対称マルチプロセッシング）の論理空間を完全に再現** できます。

実機ボードの完成を待つことなく、コア間通信（OpenAMP等）や、FPGAを通じた複雑な3者間ハンドシェイクのプロトコル結合テストを、PC上の論理空間で先行して完了できる強力なアジャイル開発プラットフォームとなります。

## 5. コア間通信（OpenAMP / RPMsg）の物理構造とエミュレーション設計

Zynq、i.MX 95、PolarFire SoCのような、アーキテクチャや役割が異なる複数のCPUコア（Cortex-A、Cortex-M、RISC-Vなど）を1つのチップに混載したSoC（ヘテロジニアス・マルチコアSoC）において、コア同士がデータをやり取りする仕組みは、主に**「物理レイヤー（ハードウェアの通信路）」**と、その上で動く**「ソフトウェアレイヤー（通信プロトコル）」**の2つのレイヤーの組み合わせで成り立っています。

### 5.1. 物理レイヤー（ハードウェアの通信路）

コア間の通信は、パケットのようにデータを飛ばしているわけではなく、基本的には「同じメモリ空間を共有し、お互いにシグナルを送り合う」という構造をとります。主に使用されるコンポーネントは以下の3つです。

1. **インターコネクト（高速バスネットワーク）**:
   各SoCの内側には、AMBA（AXI4、ACE、CHI）などの超高速なバスネットワーク（インターコネクト）が張り巡らされています。
   * **Zynq**: ARMの「NIC-400」や独自クロスバー。
   * **i.MX 95**: 最新の「FlexNoC（Network-on-Chip）」を採用。
   * **PolarFire SoC**: 各RISC-Vコア（U54/E51）を繋ぐ高速コヒーレントスイッチ。
   このバスを介して、異なるコア（例：Cortex-AとCortex-M）が同じ内蔵SRAMや外部DDRメモリに同時にアクセスできるようになっています。

2. **共有メモリ（Shared Memory / メモリアドレスの共有）**:
   最も原始的かつ高速なデータ転送路です。メモリ空間の一部を「コアAもコアBも読み書きできる共有領域」として確保します。コアAがデータを共有メモリに書き込み、コアBがそれを読み出すことで、大容量のデータ（カメラ画像、通信パケットなど）をゼロコピーで転送します。

3. **内部割り込み（IPM: Inter-Processor Mechanism / メッセージングハードウェア）**:
   共有メモリにデータを置いただけでなく、相手のコアに通知するための**物理的な割り込み専用ハードウェア**がSoC内部に用意されています。
   * **Zynq**: **SGI (Software Generated Interrupt)** を使用。GIC（割り込みコントローラ）を介して相手コアのピンを仮想的に叩きます。
   * **i.MX 95**: **MU (Messaging Unit)** という専用IPを搭載。レジスタに1ビット書き込むだけで、相手コアにハードウェア割り込みを飛ばせます。
   * **PolarFire SoC**: **PLIC (Platform-Level Interrupt Controller)** およびコア間のダイレクト割り込み（IPI: Inter-Processor Interrupt）線を使用します。

### 5.2. ソフトウェアレイヤー（通信プロトコル：OpenAMP / RPMsg）

この「共有メモリ」と「内部割り込み」というハードウェアを利用して、OS間（LinuxとRTOS、あるいはベアメタルなど）でクリーンに通信を行うための標準仕様が **RPMsg (Remote Processor Messaging) / OpenAMP** です。

#### データの流れる仕組み（制御フロー）
1. **バッファの初期化**: 共有メモリ上に、`vring`（仮想リングバッファ）と呼ばれる送信用・受信用のキュー構造を2つ作ります。
2. **データの書き込み**: コアA（Linux）が、`vring` の空きスロットにデータを書き込みます。
3. **割り込み通知（Kick）**: コアAはSoCのメッセージングハードウェア（MUやSGI）を叩き、コアB（RTOS）に割り込みを発生させます。
4. **データの読み出し**: コアBは割り込みハンドラ内で `vring` からデータを取り出し、処理を行います。
5. **返却通知**: コアBは処理が終わったらスロットを空にし、再度割り込みでコアAに「使い終わった」と通知します。

Linux側からは単なる「TTYデバイス（`/dev/ttyRPMSG0`）」や「ネットワークソケット」のように見え、RTOS側からは「API関数（`rpmsg_send()` など）」を叩くだけで、背後の割り込み制御やメモリ空間の不整合を意識せずに双方向通信ができる仕組みです。

### 5.3. SoC特有の「交通整理（同期とコヒーレンシ）」の注意点

ヘテロジニアスSoCで最もバグの原因になりやすいのが、**「キャッシュコヒーレンシ（メモリの同期）」の欠落**です。

* **同一クラスター内（Cortex-A同士など）**:
  ハードウェア（SCUやDSUなどのキャッシュコヒーレンスユニット）が自動でキャッシュの同期をとるため、ソフトウェア側で意識する必要はありません。
* **異なる非対称コア間（Cortex-A と Cortex-M / RISC-Vの別種類間）**:
  **ハードウェアによる自動キャッシュ同期が効きません。**
  例えば、Cortex-A55が共有メモリにデータを書き込んでも、データが高速なL1/L2キャッシュに留まっている場合、物理DDRメモリにはまだ書き込まれていません。その状態でCortex-M7がDDRメモリを読みに行くと、古い（ゴミの）データを取得してしまいます。

そのため、HALやRPMsgの内部ドライバ実装においては、ハードウェアを跨いでデータを渡す直前に、明示的に以下のような低レイヤー命令を実行して交通整理（キャッシュのフラッシュとバリア）を行う必要があります。

```cpp
// 1. 共有メモリにデータを書き込む
store_data_to_shared_memory(buffer);

// 2. キャッシュラインを強制的にDDRメモリへ押し出す（フラッシュ/クリーン）
// SoCやコンパイラ、コアのアーキテクチャに応じた命令（例：ARMのXilinx/NXP提供API）
Xil_DCacheFlushRange((INTPTR)buffer, size); 

// 3. メモリバリア命令（CPUの実行順序追い越しを防止し、完全にメモリ転送終わるのを待つ）
asm volatile("dsb sy" : : : "memory");

// 4. ここで初めて相手コアへMU（Messaging Unit）を通じて割り込みを打つ
trigger_hardware_interrupt();
```

### 5.4. F-BB検証環境におけるエミュレーションの設計思想

このようなマルチコア間通信を行うファームウェアのロジックを、実機を使わずにホストPC上でシミュレーション・検証する場合、物理的な「MUのレジスタ」や「L1/L2キャッシュの挙動」をVerilator等で生真面目に波形再現しようとすると、シミュレーションが極めて重くなります。

したがって、F-BB上のシミュレーションでは、テストシナリオ配下で以下のようにマッピングを「翻訳」して抽象化する設計が最もスマートです。

| 構成要素 | 実機環境（Zynq / i.MX 95等） | ホストPC / WSL2（F-BB検証環境） |
| --- | --- | --- |
| **トランスポート** | 物理共有メモリ（SRAM / DDR） | プロセス間共有メモリ（`shm_open` / `mmap`） |
| **シグナリング** | 内部ハードウェア割り込み（MU / SGI） | POSIXシグナル、または `pthread_mutex` / `condition_variable` |
| **データ構造** | RPMsg（`vring` バッファ） | Linux標準のパイプ（`pipe`）やリングバッファ構造 |

このように、F-BBが提供するマルチプロセス・マルチスレッド実行環境上で、ユーザーがテストシナリオ配下に「実機とホストの通信バッファの抽象化マニフェスト（ダミーRPMsg / UNIXドメインソケットラッパー等）」を配置・ビルドすることで、ファームウェアの論理的な通信シーケンスの正しさをホストPC上で高速に検証することが可能になります。
